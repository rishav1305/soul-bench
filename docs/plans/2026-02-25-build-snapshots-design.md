# Build Snapshots — Design

Date: 2026-02-25
Status: Approved

## Problem

The SOCIAL block (Block 3) needs rich material from BUILD work to create compelling content. Currently, the only structured artifact is a 1-line content-log entry and raw git diffs. Design decisions, test progression (red to green), metrics, and terminal output are lost by the time content is drafted.

## Solution

A standalone `snapshot` skill that captures structured text + terminal screenshots at key build milestones. Called automatically from block-execution at 5 points, and available manually via `/snapshot`.

## Snapshot Artifact

Each capture produces a directory: `docs/snapshots/<date>-<topic>/`

Contents:
- `snapshot.md` — Structured markdown with all milestone entries
- `*.png` — Terminal screenshots at each milestone

### snapshot.md Format

Each milestone appends a section:

```markdown
## <milestone-name>
- **Time:** HH:MM
- **Files changed:** <list from git diff --stat>
- **LOC delta:** +X / -Y
- **Tests:** N total, N passed, N failed
- **Notes:** <auto-generated summary + user notes>
```

## 5 Milestone Capture Points

| After Step | Milestone | Captures |
|------------|-----------|----------|
| 1. PLAN | `design-approved` | Design summary, approach, key files identified |
| 4. TDD | `tests-red` | Test file names, test count, screenshot of failing tests |
| 5. EXECUTE | `tests-green` | Files changed, LOC delta, pass count, screenshot of green |
| 8. SECURITY | `security-clear` | Audit findings or clean, fixes applied |
| 10. COMMIT | `shipped` | Final stats, commit hash, screenshot of final run |

## Content-log Integration

Extended format with optional snapshot path:

```
DATE | BLOCK | What happened | Content type hint | snapshot: <path>
```

The content-architect agent reads snapshot directories when producing briefs.

## Skill Interface

Inputs:
- `milestone` (required): design-approved, tests-red, tests-green, security-clear, shipped
- `topic` (required): kebab-case name for the snapshot directory
- `notes` (optional): free-text context

Actions:
1. Create `docs/snapshots/<date>-<topic>/` if not exists
2. Run `git diff --stat` to capture file/LOC changes
3. Run test suite and capture counts (if applicable to milestone)
4. Write/append to `snapshot.md`
5. Take terminal screenshot, save to directory
6. Append/update content-log entry with snapshot path

## Changes to block-execution

Add snapshot invocations at 5 existing checkpoints (no new steps). Each is a lightweight haiku call to the snapshot skill. The topic is derived from the block's task name.

## Changes to content-architect

Update the agent to:
1. Check `docs/snapshots/` for recent directories (last 72h)
2. Read snapshot.md files for structured facts and metrics
3. Include screenshot paths in the content brief for writer agents to reference
